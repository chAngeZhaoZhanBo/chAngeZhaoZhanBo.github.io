    《设计模式》其一：策略模式#

    当提到OOP时，你会想到什么呢？大多数人想到的是抽象、封装、多态与继承。

    掌握了这些OO基础，你从理论上来讲就能写出不错的OO系统了。然而，正如“即便你把C++抑或是其它编程语言的语法层面的东西学习得差不多了，你也很难写出良好的、可维护的、高效率的代码，而还是要去了解各种最佳实践”一样，在学习OO的过程中，也有必要了解设计模式。设计模式是经过总结后的OO系统设计的隐含经验，使用它们可以帮助你更好地构建OO系统，同时降低你与合作者的沟通成本。

    任何的学习都最好不要生搬硬套、死记硬背，设计模式也是一样。《设计模式》中涉及了非常多的设计模式，简单地记忆它们，不出几天可能就会忘个精光，也可能会混淆不同模式的使用场景。因此，有必要梳理不同的设计模式根本的出发点，他们解决了那些需要解决的问题。这样即便你忘记了这个设计模式的名字，你的思想也已经变成了它的形状了（雾）

    本系列用于梳理和记录我在学习《设计模式》时的一些思考，欢迎大家批评与指正orz本文将谈谈第一个设计模式——策略模式

    考虑一个基类Base和它的一族子类。Base中有一个成员函数f，那么f可能涉及到哪些OO的基础元素呢？都有哪些情况呢？

    • 所有子类全部应该有相同的行为f，我们将这个行为提取出来，放在基类Base中作为成员函数，供子类继承，以达成代码复用的目的。（如果你确信从设计的根本逻辑上，所有子类都应该有相同的行为f，也即是说后续加入的子类也均遵循行为f，则可以把f设为final，防止子类覆盖f）
    • 所有的子类全部有相似的行为f，它们可以有相同的对外接口，但内部实现却**各不相同**。此时你可能希望利用多态，基类弄一个虚函数。如果仍然存在一个“默认行为”的话，可以在基类中弄一个默认的实现；否则可以把f在基类中搞成纯虚函数。

    可能你注意到了我上文中加黑的“各不相同”，其实这并不是必要的。但如果你不能保证所有的子类都对f有不同的实现，你可能就需要“策略模式”了~

    设想一下下面的情景：Base有高达114514个子类，但这些子类对于f的实现只有4种，每种实现所占的子类数量都超过25000个。这意味着什么呢？意味着对于同样的一种实现，你需要重复编写它25000次。你可能会说，这算什么，你当Ctrl +C/V是闹着玩的吗？那再设想一下，之后你需要对某一种实现做出修改，那你每次修改都要粘贴25000次，那还不如直接被先辈雷普了呢（

    归根结底，这个问题的原因很简单，你没有代码复用啊。从OO的角度来说，不同子类对f的实现，即便内容完全相同，但在OO体系里，仍然被视为“独立的、不同的实现”。

    所以你要做的事情也很简单，寻找一种代码复用的方式，把f的每一种实现都封装成一个可复用的组件。

    所以我们正式引出策略模式：定义算法族，分别封装起来，使他们之间可以互相替换，此模式让算法的变化独立于使用算法的客户。

    C++程序员可能在想，这个简单啊，最基本的代码复用方式不就是攒个新函数出来吗？我对三种实现攒三个新函数，然后把函数指针按需传给不同的子类不就好了？

    但是不是啥编程语言都能直接操作函数指针的，何况《设计模式》这本书是针对java写的，这太不OO了，I‘m angry！

    其实我们有更OO的实现方式：

    public interface FBehavior {  public void f();}public abstract class Base {  FBehavior fb;  public void performF() {    fb.f();   }}

    对于不同的f的实现，我们只需要implement FBehavior弄出不同的class就好了。在Base的子类的构造函数中，我们按照需要构造不同的实现了FBehavior的class，即可达到策略模式的目的。

    策略模式是如此的自然，因此被《设计模式》放在最开始讲解。思考它的出现原因，可以让我们对OO和常见编程思想有更深刻的理解。
