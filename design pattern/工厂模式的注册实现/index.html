    工厂模式的注册实现#

    场景#

    最近python项目里遇到了这样的需求

    graph TD
A["User Input Arg"]-->B["A_class_factory"]
A-->C["B_class_factory"]
B-->D["A_subclass"]
C-->E["B_subclass"]
D-->H["C_aggregation"]
E-->H
H-->F
G["User Input Data"]-->F
A-.->G

    有很多种不同的User Input Data，不同格式的Data需要不同的处理逻辑。用户通过输入的命令行参数User Input Arg（后简称arg）来告知模块内部应该采用哪种处理逻辑。

    处理数据的class即为上图的C_aggregation，但实际处理逻辑分为多个子逻辑，我们用抽象类A_class和B_class抽象两个子逻辑接口，然后对于每一种arg，我们编写相应的子类表示具体逻辑。在运行时，我们通过arg获取对应的子类实例作为“组件”拼到C_aggregation中，而通过arg获取具体子类的实例正是工厂模式的常用场景，大概的代码如下所示。

    class C_aggregation:
    def __init__(self, arg:str)
        self.A = A_class_factory.getA(arg)
        self.B = B_class_factory.getB(arg)
    def process(self, data):
        data = self.A.process(data)
        data = self.B.process(data)
        return data

    没那么完美：散落各处的硬编码#

    上面的场景看上去其实很简单，但每当开发者发现有一种新的数据格式需要被处理时，他需要修改代码的多个部分

    1. 首先，编写新的A_subclass和B_subclass

    2. 为这种数据选定一个新的arg

    3. 在A_class_factory和B_class_factory中添加新的if分支（并且还要import刚刚编写好的新子类）

    4. 将新arg添加到你的python主脚本命令行参数的choices列表中

    上面四点里，第一点和第二点是无论如何都无法绕过的，但后面两点显得比较繁琐，涉及到的文件也比较多。虽然可以在doc for developer里面把添加新数据格式所需要的全部流程列好，但这种“流程”麻烦且难以记忆，我们有必要将开发“流程”简洁化。

    本场景中，导致开发、维护场景繁琐的根本在于硬编码。所谓硬编码，简单来讲就是把各种“数据”写死在代码里。一个最简单的情形是代码里需要一个常量表示圆周率，暂时选定了3.14，然后把这个3.14写的到处都是，导致之后如果想要更精确的3.14159就要把所有3.14的出现都改一遍，带来极大的维护困难，并且每当有新的代码需要用到这个常量时，需要开发者始终记得当前使用的数值到底是3.14还是3.14159还是其他的什么值。

    总计一下，硬编码的问题主要是：

    • 带来极大的维护困难（很有可能涉及多处甚至多文件维护）
    • 给开发者带来了额外的记忆成本

    而在我们的场景中，硬编码的是“arg到子类的映射关系”。它可能以一堆if语句的形式存在，但它的本质是映射。而这个映射的维护是相对独立于子类本身的，开发者在编写子类时要去其他的多个源文件做出修改，我们希望把开发者从这个负担中解放出来。

    注册器#

    于是我们选择了注册器模式。

    这个实现的核心思路如下：在定义某一种新的子类时，为其添加装饰器，该装饰器不改变类的行为，单纯将其“注册”到某个“词典”里，词典是arg->class的，这样工厂类中就不需要写大量if分支，只需要用arg作为key得到对应的class即可，choices也无需手写而采用这个词典的keys即可。

    我们来看一下代码的具体实现

    class Register:

    def __init__(self, registry_name):
        self._dict = {}
        self._name = registry_name

    def __setitem__(self, key, value):
        if not callable(value):
            raise Exception(f"Value of a Registry must be a callable!\nValue: {value}")
        if key in self._dict:
            logging.warning("Key %s already in registry %s." % (key, self._name))
        self._dict[key] = value

    def register(self, target):

        def add(key, value):
            self[key] = value
            return value

        # @reg.register('alias')
        return lambda x: add(target, x)

    def __getitem__(self, key):
        return self._dict[key]

    def __contains__(self, key):
        return key in self._dict

    def keys(self):
        """key"""
        return self._dict.keys()


class Registers:

    def __init__(self):
        raise RuntimeError("Registries is not intended to be instantiated")

    A = Register("A")
    B = Register("B")

    在上面的实现里，Registers掌管所有种类的Register实例，每一种Register都可以通过register接口将arg->class的映射注册到自己的内部_dict里，具体使用方式类似下面的代码

    @Registers.A.register("New_A")
class New_A_subclass:
    """do something"""


"""another python file"""
class A_class_factory:
    def get_A(self, arg):
        return Registers.A[arg]()

"""another python file"""
choices = Registers.A.keys()

    通过这种方式，开发者只需要在编写的子类前加上“注册语句”，就能让一切按照自己预想的方式工作。开发流程被局限在了子类所在的文件中，无需去其他的源文件中做出相应的修改，大大减少了开发负担。

    一点小问题：导入模块#

    当然，由于python的机制，我们需要import使用了注册机制的模块，才能使得一个类真正地被注册。

    因此，一个依赖“注册结果”的模块（比如工厂类所在的模块，工厂类需要注册结果来生产arg对应的类），必须要import所有我们所需要的注册的类所在的模块，而这不又是一种硬编码吗（import的模块列表依然需要手动维护）

    好在，python有importlib，可以动态地import模块。因此我们可以为Register维护需要注册的模块路径，封装一个函数import_register_modules()，这个函数使用importlib动态导入你指定的模块路径的集合。在代码入口处调用import_register_modules。这样就把到处都可能存在的硬编码集成到了import_register_modules里，每当添加了新的子类，只需要修改这一处就能解决import问题。

    使用了注册机制后，开发者添加新子类要做的事情是：

    1. 首先，编写新的A_subclass和B_subclass
    2. 为这种数据选定一个新的arg
    3. 给新编写的子类添加“注册语句”
    4. 修改import_register_modules

    虽然看上去依然是四点，但减轻的负担谁用谁知道.jpg

    C++实现#

    毕竟我是一个C++ boy，尽管业务场景里要求的语言是python，但还是也要研究一下C++中该怎么实现这一模式。

    python实现中，我们直接建立了arg到类名的映射。而在C++中我们不能简单地这么做，归根结底，原因在于python中的类名其实是一个callable，可以调用它来创建对应的对象实例，而C++中的类名则没有这么好的事情了（

    那么，首先我们要解决的子问题就是如何通过类名获取一个callable可以创建对象。这个可以通过模板解决

    而Register类中维护arg到Base*()的映射

    class Register {
    static std::map<std::string, std::function<Base*()>>& getDict() {
       static std::map<std::string, std::function<Base*()>> _dict; 
       return _dict;
    }
    public:
    static std::function<Base*()>& operator[](const std::string& str) {
        return getDict()[str];
    }
    static std::vector<std::string> keys() {
        std::vector<std::string> result;
        std::transform(getDict().begin(), getDict().end(), back_inserter(result),
                       [](auto const& _pair) {return _pair.first;});
        return result;
    }
    template<typename T>
    static void register(const std::string& key) {
        getDict().emplace(key, []{return new T;});
    }
};

    为了让注册语句简洁，不要过多暴露Register的实现细节，我们使用宏

    #define REGISTER_BASE(T, key) Register::register<T>(key);
