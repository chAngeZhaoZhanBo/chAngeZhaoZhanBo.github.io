    最大排列#

    1. 引子#

    之所以要写这篇文章，是因为看到了这样的一道算法题：

    假设有n个正整数，将他们连成一片会得到一个新的大整数，请求出能组成的最大和最小整数。

    如，我们有四个正整数23,9,182,79，连成的最大整数时97923182，最小是18223799

    按词典序排序显然是不行的（如输入是81和8，8的字典序更小，但881比818大）

    在思考这个问题时，我突然意识到这是一类问题的特例，因此试图写一篇文章把这类问题明确定义下来，并给出解决方案。

    2. 问题形式化#

    给定一个序列[x_1, x_2, ..., x_n]，它的所有排列构成集合P=\{P_1, P_2, ..., P_{n!}\}，存在一个值映射f:P \rightarrow V

    V是一个有**全序关系**的有限集合，自然就有最大值和最小值

    对于上面的问题，f就是“连接”的函数，V就是正整数集合的某个子集，全序关系就是正整数的大小关系

    3. 排列与交换#

    我们知道，同一个序列的两种不同的排列，可以仅通过不断交换某对相邻元素，就可以相互转换。

    **证明：**冒泡排序是仅有交换相邻元素操作的排序，而任意的排列都可以通过冒泡排序变为“升序排列”，同理任意的排列都可以从“升序排列”出发仅通过交换相邻元素得到，则原命题显然得证。（A->B可化归为A->升序排列->B）

    而对于P中的某个P_i，交换其内部的某个相邻对之后，所对应的V值无非就是变大、变小和不变。

    一个很直观的想法是，如果我们不断交换相邻对，保证每次交换都让V值变大/变小，最后就能收敛到最大/最小值。

    但直观归直观，很容易就能构造出反例：

    min = f(1, 2, 3) < ... < f(2, 1, 3) < f(1, 3, 2) = max

    如果我们从排列[1, 2, 3]出发，选择交换1和2，值的确变大了，但随后再进行任何相邻项交换都会变小。而f(2, 1, 3)并不是最大值，f(1, 3, 2)更大，只是[2, 1, 3]不能通过一次交换相邻项到达[1, 3, 2]而已。

    这个思路无法解决这一大类抽象问题，那它能否解决这一大类问题的某个子集呢？通过添加一些其他限制条件。

    考虑到上面的反例，我们可以加上这样的限制：如果一个排列不能通过交换相邻项使得V值变大，则它必须是V值最大的那个排列，这保证了上面的过程能收敛且一定收敛在最大值

    4. 可行性考量#

    看上去我们很好的解决了我们所形式化的问题**的一个子集**，但我们有更重要的可行性问题需要解决：

    4.1 计算复杂度#

    如果没有收敛，我们如何找到需要交换的相邻项；如果收敛到最大排列，我们如何确定自己此时在最大排列上。

    如果没有任何额外信息，我们能做的只有：遍历n-1种交换相邻项方案，算出n-1个排列的V值以及自己当前的V值（计算n次f）并进行最多n-1次比较。而两个排列之间最多n(n-1)/2次交换就可以互相到达。

    不管怎么说，这比计算全部n!种排列的V值，然后再找到最大、最小值要好多了

    4.2 如何验证一个问题属于我们能处理的范围#

    **“如果一个排列不能通过交换相邻项使得V值变大，则它必须是V值最大的那个排列”**这个条件过于抽象

    我们很难看出一个问题是否满足这一条件
